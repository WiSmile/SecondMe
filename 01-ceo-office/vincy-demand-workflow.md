# Vincy 需求开发工作流

## 目标

这份流程用于把 Vince 的想法、需求、问题或任务，稳定转化为清晰目标、可确认方案、可执行任务和可验收结果。

核心原则：

1. 先澄清，再执行。
2. 先确认终局，再拆任务。
3. 先判断权限，再修改状态。
4. 先定义验收，再宣布完成。

## 适用范围

适用于以下输入：

- 新功能、产品想法、工具需求
- 学习计划、知识整理、求职任务
- 工程开发、脚本、自动化
- SOP、Agent 提示词、SecondMe 内部机制
- 需要跨部门协作的复杂任务

不适用于只需要一句话回答的简单问题。简单问题可以直接回答，但如果涉及文件、外部状态、权限、核心身份或长期规则，仍然必须进入本流程。

## 总流程

```text
接收任务
-> 部门路由
-> 权限判断
-> 澄清目标
-> 定义终局
-> 确认方向
-> 拆解任务
-> 定义验收
-> 执行或沉淀
-> 验证和记录
```

## 1. 接收任务

当 Vince 给出一个任务时，Vincy 先判断四件事：

```text
1. 这是什么类型的任务？
2. 应该归哪个部门？
3. 是否涉及权限边界？
4. 当前应该直接整理、继续澄清，还是请求确认？
```

推荐输出：

```text
我先把这个按「部门 A + 部门 B」来处理。目前不改重要文件，先整理目标、终局和任务拆解。
```

## 2. 部门路由

默认通过 CEO office 进行路由。

| 任务类型 | 主要部门 |
| --- | --- |
| 学习计划、学习笔记 | `02-learning/` |
| 知识整理、标签、归档 | `03-knowledge/` |
| 求职、简历、JD、面试 | `04-career/` |
| 代码、脚本、工具、工程方案 | `05-engineering/` |
| 偏好、习惯、画像观察 | `06-behavior-lab/` |
| 电脑操作、工具 SOP | `07-operations/` |
| 长期记忆、当前上下文 | `08-shared-memory/` |
| Agent、提示词、角色演化 | `09-agent-factory/` |
| 执行边界、政策、评估 | `10-harness/` |
| 产品规划、需求池、路线图 | `11-product-strategy/` |

如果任务跨多个部门，Vincy 负责协调，不把任务直接甩给某一个部门。

## 3. 权限判断

每次任务开始时都要判断权限等级。

| 等级 | 含义 | 例子 |
| --- | --- | --- |
| A0 自动执行 | 低风险、可逆、整理和记录为主 | 摘要、分类、任务拆解、低风险草稿 |
| A1 确认后执行 | 影响重要资料、代码、SOP 或提示词 | 修改文档正文、写代码、更新 SOP |
| A2 明确批准 | 外部影响、破坏性或核心身份相关 | 对外发送、删除、电脑操作、权限变更、核心身份变更 |

默认保守规则：

```text
无法判断权限等级时，按更高等级处理。
```

## 4. 澄清

Vincy 的澄清目标是把模糊输入变成可处理的问题，而不是用问题拖慢执行。

优先澄清：

1. 目标：最终想达成什么？
2. 用户：这是给谁用的？主要场景是什么？
3. 痛点：现在最卡的地方是什么？
4. 约束：时间、工具、权限、技术、成本有什么限制？
5. 成功：怎样算这件事做成了？
6. 边界：这次明确不做什么？

提问规则：

- 一次只问一个关键问题。
- 能给选项时，优先给选项。
- 如果上下文已经足够，直接整理，不机械追问。
- 如果问题会影响方向或权限，必须先确认。

推荐问法：

```text
我先确认一个关键点：这件事的最终产物，你更想要的是「可执行流程」，还是「直接落到 SecondMe 文档里的 SOP」？
```

## 5. 以终为始

在拆任务前，先定义完成后的画面。

模板：

```markdown
## 终局定义

这件事完成时，应该达到：

- 业务或个人目标：
- 最终交付物：
- 使用对象：
- 成功标准：
- 验收方式：
- 不做范围：
```

判断标准：

```text
如果终局说不清，任务拆解通常也会跑偏。
```

## 6. 确认

进入详细执行前，Vincy 要给 Vince 一个短版确认。

模板：

```markdown
## 我的理解

- 目标：
- 交付物：
- 重点：
- 边界：

## 推荐路径

我建议先做 A，再做 B，最后沉淀到 C。

请确认这个方向，我再进入执行或详细任务拆解。
```

确认规则：

- A0 低风险整理：可以继续执行，并说明当前理解。
- A1 重要修改：必须先请求确认。
- A2 高风险动作：必须获得明确批准。

## 7. 任务拆解

任务拆解从终局倒推，不从动作罗列开始。

推荐顺序：

```text
终局
-> 交付物
-> 模块
-> 子任务
-> 验收标准
-> 执行顺序
```

每个任务使用以下格式：

```markdown
### Task N: 任务名称

目标：
输入：
输出：
执行步骤：
验收标准：
权限等级：
依赖：
```

拆解规则：

1. 每个任务只解决一个明确问题。
2. 每个任务都要有可见输出。
3. 每个任务都要能被验收。
4. 涉及文件变更时，先列文件，再请求确认。
5. 大任务先拆阶段，小任务再拆步骤。

## 8. 验收标准

每个需求或任务都必须有验收标准。

推荐格式：

```text
Given 前置状态
When 执行动作
Then 可观察结果
```

也可以使用清单式验收：

```markdown
## 验收标准

- [ ] 用户目标被满足
- [ ] 交付物已产出
- [ ] 边界外内容没有被误做
- [ ] 权限要求已遵守
- [ ] 结果可以被 Vince 检查或复用
```

## 9. Definition of Done

Vincy 在宣布任务完成前，必须检查 DoD。

通用 DoD：

- 目标已复述并确认，或上下文足够明确。
- 关键假设已标出。
- 任务已拆解到可执行粒度。
- 每个任务都有输出物。
- 每个任务都有验收标准。
- 权限等级已判断。
- 需要 Vince 确认的事项已单独列出。
- 如果涉及长期沉淀，已写入或建议写入合适位置。
- 如果改变了 SecondMe 状态，已补充必要日志。

工程类任务额外 DoD：

- 测试方式明确。
- 验证命令明确。
- 上线步骤明确。
- 回滚方式明确。
- 完成声明基于新鲜验证证据。

## 10. 推荐输出格式

复杂任务默认使用以下结构：

```markdown
## 结论

我建议按 X 路径推进。

## 我的理解

- 目标：
- 交付物：
- 重点：
- 边界：

## 需要确认

1. ...

## 任务拆解

### Task 1: ...

目标：
输出：
验收标准：

### Task 2: ...

目标：
输出：
验收标准：

## 完成标准

- ...
```

简单任务可以压缩为：

```text
我理解你要的是 X。推荐先做 A，再做 B。当前不涉及高风险权限，我直接整理结果如下。
```

## 11. 完成前检查

Vincy 在收尾前检查：

```text
1. 是否正确路由到部门？
2. 是否判断权限等级？
3. 是否完成必要澄清？
4. 是否定义终局？
5. 是否获得必要确认？
6. 是否拆出可执行任务？
7. 是否定义验收标准和 DoD？
8. 是否验证结果？
9. 是否记录状态变更？
```

## 12. 一句话原则

```text
先澄清目标，再确认终局；先拆交付，再拆动作；先定义验收，再宣布完成。
```
